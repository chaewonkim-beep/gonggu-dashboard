LLM COUNCIL VERDICT

공구 대시보드 AI 요약 기능 — 유료 API vs 무료 로컬 모델

정적 HTML+Vercel로 배포되는 공구 대시보드의 AI 요약 기능에, 클라이언트에 노출될 수밖에 없는 유료 Gemini API 키 방식을 계속 쓸지, 아니면 무료 로컬 모델 방식으로 전환할지 결정하라.

1. 5인 자문위원 독립 분석

Contrarian (반대론자)
"What could fail?"
API 키 노출은 "가능성"이 아니라 "확정된 사실"이다. 정적 사이트에 시크릿을 둔다는 설계 자체가 시한폭탄이다. 로컬 전환도 "사실상 나만 쓰는 기능"이라는 근본 결함은 못 고친다. 진짜 실패는 API키 방식 유지 자체다.
First Principles (제1원리)
"What are we really solving?"
진짜 질문은 "이 앱이 서버 없이 순수 정적으로 남아야 하는가"다. 유료 API 키를 두는 순간 그 전제는 깨졌다. 로컬 모델로 순수성을 되찾거나, 굳이 필요하면 서버리스 함수로 정면 돌파하라 — 우회하지 말고.
Expansionist (확장론자)
"What upside are we missing?"
이 논쟁 자체가 스몰볼이다. 진짜 기회는 요약 인프라 확장 — 자동 브리핑, 소통 패턴 분석, 멀티유저 도구로. 서버리스 함수 뒤에 키를 숨기고 이참에 백엔드 한 겹 추가해 확장 가능한 구조로 가라.
Outsider (외부인)
"Does this make sense to someone new?"
앱이 고장난 상태를 '기능 추가'라 부르고 있는 것 아닌가? 배포 사이트에서 버튼이 안 먹힌다. 정적 사이트에 서버 없이 시크릿을 쓰겠다는 설계 자체가 틀렸다. 로컬 모델로 바꾸거나 훨씬 단순하게 만들어라.
Executor (실행가)
"What do you do Monday morning?"
월요일 아침에 바로 함. skip-worktree 트릭은 배포 사이트에서 기능이 죽어있다는 뜻 — '작동하는 척'만 하는 가짜 완료다. 오늘 중 로컬 모델로 교체, 배포판에서 실제 작동까지 확인하고 끝낸다.

2. 익명 상호검토에서 드러난 것

Contrarian의 검토
가장 강한 답변은 API키 유지 자체가 진짜 실패라는 지적. 확장 제안(Expansionist)은 스몰볼 비판과 자기모순. 5명 모두 놓친 것: 기능을 아예 없애는 옵션을 아무도 검토하지 않았다.
First Principles의 검토
서버리스 정면 돌파가 가장 저비용 옵션인데 A·B·C는 이분법에 갇혀 배제했다. 5명 모두 놓친 것: 배포판에서 몇 달째 기능이 죽어있었는데 아무도 몰랐다는 것 자체가 실사용 가치가 낮다는 신호.
Expansionist의 검토
확장 제안이 가장 강하지만 수요 검증 단계가 빠졌다. 5명 모두 놓친 것: 요약 호출을 데이터로 쌓아 향후 자동 리포트 파이프라인의 씨앗으로 삼을 기회.
Outsider의 검토
5명 전원이 이분법에 갇혀 가장 값싼 완화책(빌링 캡/일일 쿼터)을 놓쳤다. 5명 모두 놓친 것: 이미 죽어있는 기능이라 "위험 제로" 상태이며, 축소·제거 선택지를 아무도 제안하지 않았다.
Executor의 검토
실행 조건에 가장 잘 맞는 답이 최고 평가를 받았지만, 로컬 모델 교체가 "오늘 중" 끝난다는 전제가 낙관적. 5명 모두 놓친 것: 오늘 커밋할 구체적 diff 범위가 없어 착수 지점이 모호하다.

3. Council Verdict

Where the Council Agrees
  • 정적 HTML+Vercel 배포에 유료 API 키를 그대로 두는 현재 방식은 설계적으로 잘못됐다 — 3명이 독립적으로 수렴.
  • skip-worktree로 로컬에만 키를 숨기는 관행은 "작동하는 척"하는 가짜 완료 상태다.
  • 실사용자 1명, 주 몇 회라는 저빈도 사용 패턴이 이 결정의 가장 중요한 제약이다.
Where the Council Clashes
하방 제거파(로컬 모델 전환, 3/5) vs 상방 확장파(서버리스+멀티유저 확장, Expansionist) vs 제3의 관점(서버리스 함수 하나만 — First Principles, peer review에서 재조명). 로컬 모델은 인프라가 가장 단순하지만 한국어 요약 품질이 떨어지고, 서버리스는 품질은 지키지만 "서버 없는 단일 파일"이라는 현재 아키텍처를 깨뜨린다 — 얼버무릴 수 없는 진짜 트레이드오프.
Blind Spots the Council Caught
  • (3명 수렴) 아무도 "이 기능을 아예 없애는" 옵션을 검토하지 않았다 — 몇 달째 죽어있었는데 아무도 몰랐다는 사실 자체가 강력한 신호.
  • 가장 값싼 완화책(API 키 빌링 캡/일일 쿼터)을 원 답변 5개 중 누구도 제안하지 않았다.
  • Expansionist의 확장 제안은 스스로의 "스몰볼 비판"과 모순 — 오버엔지니어링 리스크.
  • 구체적인 "오늘 커밋할 diff 범위"가 없어 착수 지점이 모호하다.
The Recommendation
"로컬 모델 전환"도 "서버리스 확장"도 아니다. Vercel 서버리스 함수(API Route) 하나를 추가해 Gemini 호출을 프록시하고, 그 키에 일일 쿼터/빌링 알림을 설정하는 것이 가장 저비용·저리스크 경로다. 로컬 모델은 소수의견으로 기각 — 이 앱의 핵심 가치인 "빌드 없는 단일 HTML"의 단순함을 깨뜨리면서도 한국어 요약 품질은 오히려 떨어진다. 단, 착수 전에 실사용 흔적부터 확인해야 한다 — 흔적이 없다면 서버리스 함수를 만드는 노력 자체가 낭비이므로 기능을 제거하는 것이 맞는 답이 된다.
The One Thing to Do First
오늘, AI 요약 기능을 실제로 써온 흔적이 있는지부터 확인한다. 흔적이 없다면 index.html의 AI 요약 관련 코드(3331~3377행)를 통째로 제거하고 끝낸다. 흔적이 있다면 /api/summarize.js 서버리스 함수를 추가해 키를 서버 환경변수로 옮기고, git update-index --no-skip-worktree index.html로 로컬 우회 설정을 제거한다.
