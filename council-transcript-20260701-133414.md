# LLM Council — 의사결정 전체 기록

**일시:** 2026-07-01 13:34
**원 질문:** 공구 대시보드에 AI 요약 기능을 유료 API로 계속 쓸지, 아니면 무료 로컬 모델로 바꿀지

## 프레이밍된 질문

정적 HTML+Vercel로 배포되는 공구 대시보드의 AI 요약 기능에, 클라이언트에 노출될 수밖에 없는 유료 Gemini API 키 방식을 계속 쓸지, 아니면 무료 로컬 모델 방식으로 전환할지 결정하라.

## 배경 컨텍스트 (스캔 결과)

- `index.html`(단일 파일) 3331~3377행: Gemini 2.5 Flash API를 호출해 카카오톡/이메일 대화를 불릿 요약하는 기능.
- GitHub에 커밋된 배포판(HEAD)은 `GEMINI_API_KEY = ''`로 비어 있음 — **실제 배포 사이트에서는 AI 요약 버튼이 작동하지 않는다.**
- 로컬 작업 파일에는 실제 키 값이 들어 있고, `index.html`에 git `skip-worktree` 비트가 설정되어 있어(`git ls-files -v` 결과 `S`) 이 로컬 변경 사항이 git status/diff에 전혀 나타나지 않는다.
- 실사용자는 사실상 1명(공구 담당자), 사용 빈도는 소통 기록을 남길 때만(주 몇 회 수준).

---

## 1단계 — 5인 자문위원 독립 분석

### Contrarian (반대론자) — "What could fail?"
> API 키 노출은 "가능성"이 아니라 "확정된 사실"이다. 배포판 키가 비어 있어 지금은 작동 안 하지만, 언젠가 담당자가 편의를 위해 실키를 커밋하거나 skip-worktree 설정을 실수로 풀면 그 순간 전 세계에 키가 뿌려진다. 무료·유료 구도가 문제가 아니라 "정적 사이트에 시크릿을 둔다"는 설계 자체가 시한폭탄이다. 로컬 모델 전환은 키 유출 리스크를 원천 차단하지만, 로컬 전용이라는 건 지금과 똑같이 "사실상 나만 쓰는 기능"이라는 근본 결함은 안 고쳐진다. 진짜 실패는 API키 방식 유지 자체다.

### First Principles (제1원리) — "What are we really solving?"
> AI 요약은 부가 기능이지 핵심 문제가 아니다. 진짜 질문은 "이 앱이 서버 없이 순수 정적으로 남아야 하는가"다. 유료 API 키를 클라이언트에 두는 순간 그 전제는 이미 깨졌고, skip-worktree 우회는 근본 문제(정적 배포 모델과 서버 필요 기능의 불일치)를 감춘 임시방편일 뿐이다. 사용자 1명, 주 몇 회 사용이라면 로컬 모델로 전환해 아키텍처의 순수성을 되찾는 게 맞다. 굳이 API 키가 필요하다면 서버리스 함수 하나를 추가해 문제를 정면으로 해결하라 — 우회하지 말고.

### Expansionist (확장론자) — "What upside are we missing?"
> 지금 이 논쟁 자체가 스몰볼이다. 실사용자 1명, 주 몇 회 쓰는 기능에 '유료냐 로컬이냐'를 따지는 건 상방을 놓치는 것. 진짜 기회는 이 요약 인프라를 확장하는 것—공구 진행상황 자동 브리핑, 7개사 커뮤니케이션 패턴 분석, 나아가 다른 담당자에게도 여는 멀티유저 도구로. 키를 서버리스 함수(Vercel Edge Function) 뒤로 숨기면 보안도 해결되고 동시에 이 기능을 팀 전체 자산으로 키울 발판이 마련된다. 지금 로컬 우회로 땜빵하지 말고, 이참에 백엔드 한 겹 추가해 확장 가능한 구조로 가라.

### Outsider (외부인) — "Does this make sense to someone new?"
> 이거 그냥 앱이 고장난 상태를 '기능 추가'라고 부르고 있는 거 아닌가요? 배포된 사이트에서 버튼 눌러도 안 되는데, 사용자가 담당자 한 명뿐이라 아무도 눈치를 못 챈 것뿐입니다. skip-worktree로 로컬에만 키를 숨기는 방식은 신입 개발자가 보면 "왜 배포판과 로컬판이 다른 파일 행세를 하지?"부터 물을 구조입니다. 유료 키 노출 리스크 운운하기 전에, 애초에 정적 사이트에 서버 없이 시크릿을 쓰겠다는 설계 자체가 틀렸습니다. 주 몇 회 쓰는 개인용 기능에 이 정도 복잡한 우회책을 유지할 이유가 없습니다. 로컬 모델로 바꾸거나, 최소한 브라우저에서 직접 붙여넣은 개인 키를 localStorage에 저장하는 방식으로 단순화하세요.

### Executor (실행가) — "What do you do Monday morning?"
> 월요일 아침에 바로 함. 유료 API 키 노출 구조는 더 이상 손댈 가치 없다 — skip-worktree 트릭은 배포 사이트에서 기능이 아예 죽어있다는 뜻이고, 이건 '작동하는 척'만 하는 가짜 완료다. 사용자 1명, 주 몇 회 쓰는 기능에 서버/프록시 세우는 건 과잉이니 바로 로컬 모델(브라우저 내 transformers.js나 Ollama 연동)로 갈아탄다. 오늘 중으로 키 의존성 코드 걷어내고 로컬 요약으로 교체, 배포판에서 실제로 눌러서 되는지 확인까지 끝낸다. 완벽한 요약 품질보다 "배포 사이트에서 진짜 작동하는 것"이 최우선.

## 2단계 — 익명 상호검토 (A~E 셔플, 매핑은 최종 판결에서 공개)

셔플 매핑: A=Executor, B=Contrarian, C=Outsider, D=Expansionist, E=First Principles

### Contrarian의 검토
> ① 가장 강한 답변: B. '유료냐 로컬이냐'라는 프레임 자체를 깨고, API 키 방식 유지가 진짜 실패라고 지적하면서 로컬 전환도 근본 결함(사실상 1인용)은 못 고친다고 짚은 점이 날카롭다.
> ② 맹점: D는 스몰볼을 비판하다 곧장 멀티유저·서버리스 확장을 제안하는데, 이는 주 몇 회 쓰는 개인 기능에 오히려 더 큰 오버엔지니어링이다. 문제 규모 비판이 자기모순.
> ③ 5명 모두 놓친 것: 아무도 '이 기능을 아예 없애는 옵션'을 검토하지 않았다. 사용자 1명, 주 몇 회, 이미 배포판에서 죽어있는 기능이라면 전환 비용 자체가 낭비일 수 있는데, 모두 '계속 살린다'는 전제만 놓고 방식만 논쟁했다.

### First Principles의 검토
> ① 가장 강한 답변은 E다. "정적이냐 서버가 필요하냐"는 진짜 전제 충돌을 짚었고, 이미 Vercel을 쓰고 있으니 서버리스 함수 하나로 정면 해결하는 저비용 옵션을 버리지 않았다.
> ② 가장 큰 맹점은 A·B·C — 유료키 vs 로컬모델 이분법에 갇혀, 사실 API 라우트 한 줄 추가가 로컬 모델 전환보다 훨씬 싸고 품질도 그대로 유지된다는 선택지를 사실상 배제했다.
> ③ 5명 모두 놓친 것: 배포판에서 몇 달째 기능이 죽어 있었는데 아무도 몰랐다는 사실 자체가 '이 기능의 실사용 가치가 낮다'는 강력한 신호라는 점 — 무엇으로 고칠지 논하기 전에 '아예 없애도 되는가'를 아무도 묻지 않았다.

### Expansionist의 검토
> ① 가장 강한 답변은 D. 유일하게 "키를 어디 두느냐"를 넘어 이 요약 기능 자체를 인접 자산(멀티유저·자동 브리핑)으로 볼 상방을 짚었다.
> ② 맹점: D도 확장을 전제로만 말할 뿐, 서버리스 도입 전에 지금 요약 결과·질문 로그를 축적해 수요를 검증하자는 단계는 생략했다.
> ③ 5명 모두 "키를 어디 둘까"에만 갇혀, AI 요약 호출 자체를 데이터로 쌓아 향후 공구사별 자동 리포트·이상징후 알림 파이프라인의 씨앗으로 설계할 기회는 아무도 언급하지 않았다.

### Outsider의 검토
> ① 가장 강한 답변: B. 노출을 "가능성"이 아닌 예정된 사고로 재정의하고, 로컬 전환도 근본결함(1인 전용성)은 못 고친다고 짚은 점에서 A·E보다 한 층 더 나아감.
> ② 맹점: 5명 전원이 "로컬모델 vs API키 재설계"라는 이분법에 갇혀, 가장 값싼 완화책 — Gemini 키에 예산상한(billing cap/일일쿼터)만 걸면 유출돼도 피해가 몇 달러 수준이라는 계산을 아무도 안 함.
> ③ 놓친 것: 지금 배포판에서 기능이 이미 죽어있다는 것 자체가 사실상 "위험 제로" 상태라는 점. 주 몇 회, 1인 사용이면 아예 기능을 없애거나 로컬 전용 도구로 남기는 "축소" 선택지를 아무도 제안하지 않았다.

### Executor의 검토
> ① 가장 강한 답변: A. 실행 조건(사용자 1명, 주 몇 회)에 딱 맞게 오늘 안에 끝낼 수 있는 최소 작업을 제시하고, "배포판에서 실제로 눌러서 확인"까지 완료 기준에 넣은 게 유일하다. B·C·E는 진단은 맞지만 다음 커밋에 뭘 넣을지가 안 나온다.
> ② A의 맹점: 로컬 모델(transformers.js/Ollama) 교체가 "오늘 중"에 끝난다는 전제가 낙관적 — 모델 다운로드 크기, 브라우저 성능, Ollama라면 결국 로컬 서버라 정적 배포 원칙과 충돌하는 문제를 안 짚었다.
> ③ 5명 모두 놓친 것: "오늘 뭘 커밋할지"를 실제 코드 diff 수준으로 아무도 안 정했다. 지금 키를 무효화(revoke)할지, 어떤 라이브러리를 쓸지 등 첫 PR의 구체적 범위가 없어 논쟁만 있고 착수 지점이 없다.

## 3단계 — COUNCIL VERDICT (의장 종합 판결)

### Where the Council Agrees
1. 정적 HTML+Vercel 배포에 유료 API 키를 그대로 두는 현재 방식은 설계적으로 잘못됐다 — Contrarian·First Principles·Outsider가 각자 다른 표현("시한폭탄", "전제가 깨짐", "설계 자체가 틀렸다")으로 독립 수렴.
2. `skip-worktree`로 로컬에만 키를 숨기는 현재 관행은 "작동하는 척"하는 가짜 완료 상태다 — Executor·Outsider가 명시적으로 지적.
3. 실사용자 1명, 주 몇 회라는 저빈도 사용 패턴이 이 결정에서 가장 중요한 제약 조건이라는 데 5명 모두 동의.

### Where the Council Clashes
- **하방 제거파(Contrarian·Outsider·Executor, 원 답변 3/5):** 로컬 모델로 전환해 API 키 노출 리스크 자체를 없애야 한다.
- **상방 확장파(Expansionist):** 오히려 서버리스 함수로 키를 숨기고 이 기능을 멀티유저·자동 브리핑 자산으로 키워야 한다.
- **제3의 관점(First Principles, peer review에서 재조명):** 로컬 모델도 서버리스 확장도 아니고, Vercel에 이미 있는 인프라로 서버리스 함수 하나만 추가하면 로컬 모델 전환보다 싸고 품질 손실도 없다 — peer review에서 First Principles 자신의 원 답변(E)이 "가장 강한 답변"으로 재평가됨.
- 이 충돌은 얼버무릴 수 없는 진짜 트레이드오프다: 로컬 모델은 인프라가 가장 단순하지만 브라우저 내 한국어 요약 품질이 떨어지고, 서버리스는 품질을 지키지만 "서버 없는 단일 파일"이라는 현재 아키텍처의 단순함을 깨뜨린다.

### Blind Spots the Council Caught
1. **(3명 독립 수렴, 가장 강력한 맹점)** 아무도 "이 기능을 아예 없애는" 옵션을 검토하지 않았다. 배포판에서 기능이 몇 달째 죽어 있었는데 아무도 몰랐다는 사실 자체가, 이 기능의 실사용 가치가 애초에 낮았을 수 있다는 강력한 신호다(Contrarian, First Principles, Outsider 공통 지적).
2. 가장 값싼 완화책 — Gemini API 키에 일일 쿼터/빌링 알림(cap)을 거는 것 — 을 원 답변 5개 중 누구도 제안하지 않았다(Outsider의 검토에서만 드러남).
3. Expansionist의 확장 제안은 "스몰볼 비판"과 스스로 모순된다 — 주 몇 회 쓰는 개인 도구에 멀티유저 인프라를 얹는 것은 그 자체로 오버엔지니어링이다(Contrarian의 검토에서 드러남).
4. 5명 모두 "오늘 커밋할 구체적 diff 범위"를 정하지 않아, 논쟁은 풍부했지만 실제 착수 지점이 없다(Executor의 검토에서 드러남).

### The Recommendation
"로컬 모델로 전환"도 "서버리스로 확장"도 아니다. 가장 저비용·저리스크 경로는 **Vercel 서버리스 함수(API Route) 하나를 추가해 Gemini 호출을 프록시**하고, 그 키에 **Gemini 콘솔에서 일일 쿼터/빌링 알림을 설정**하는 것 — First Principles의 "서버리스로 정면 돌파"와 Outsider가 검토에서 짚은 "빌링 캡" 통찰을 결합한 안이다. 로컬 모델(transformers.js/Ollama)은 소수의견으로 기각한다: 이 앱의 핵심 가치는 "빌드 없는 단일 HTML 파일"인데, 브라우저 내 LLM 런타임이나 로컬 서버 연동은 그 단순함을 깨뜨리면서도 한국어 불릿 요약 품질은 오히려 떨어진다. 단, 이 작업에 들어가기 전에 3명이 수렴한 맹점부터 확인해야 한다 — 실사용 흔적이 없다면 서버리스 함수를 만드는 노력 자체가 낭비이므로, 기능을 아예 제거하는 것이 맞는 답이 된다.

### The One Thing to Do First
오늘, 이 요약 기능을 실제로 써온 흔적이 있는지부터 확인한다(예: 소통 기록 중 AI 요약을 거쳐 작성된 항목이 몇 건인지). 흔적이 없다면 `index.html`에서 AI 요약 관련 코드(3331~3377행)를 통째로 제거하고 끝낸다. 흔적이 있다면, Vercel 프로젝트에 `/api/summarize.js` 서버리스 함수를 추가해 `GEMINI_API_KEY`를 서버 환경변수로 옮기고, 클라이언트 코드는 그 엔드포인트만 호출하도록 바꾼 뒤 `git update-index --no-skip-worktree index.html`로 로컬 우회 설정과 하드코딩된 키를 제거한다.
